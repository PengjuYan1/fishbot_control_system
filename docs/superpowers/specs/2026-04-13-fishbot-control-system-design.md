# 渔场自动导航投喂机器人系统设计文档

日期：2026-04-13

Implementation notes:

- 运行时流程文档：`docs/architecture/runtime-flows.md`

## 1. 目标

构建一个新的独立工程，用于单台渔场机器人在单张地图上的自动导航投喂控制。系统需要支持：

- 地图构建、保存、加载
- 充电点设置
- 投喂点设置与排序
- 地图可视化显示
- 机器人实时位置与朝向显示
- 电量、连接、定位、导航状态监控
- 手动启动投喂任务
- 定时自动启动投喂任务
- 到达投喂点后固定停留时长
- 低电量自动回充
- 回充后行为可配置：继续任务或等待人工启动
- 兼容两种部署模式：上位机部署优先，同时支持机器人本机部署

本系统不是对旧 `ROS/ROS` 网页控制代码继续修补，而是重新建立一个结构清晰、接口边界明确的新工程。

## 2. 范围

### 2.1 第一阶段范围

- 单台机器人
- 单张地图
- 一个充电点
- 多个投喂点
- 固定停留时长投喂策略
- 地图主导型控制界面
- C++ 后端
- Web 前端 + WebSocket 实时状态推送
- ROS 适配层独立封装
- SQLite 本地配置与运行记录存储

### 2.2 暂不纳入第一阶段

- 多机器人调度
- 多地图切换与跨地图任务
- 投喂器继电器/串口/IO 联动
- 复杂任务编排
- 视觉避障或视觉识别闭环
- 云端多站点管理

系统设计时预留扩展接口，但第一阶段不实现这些内容。

## 3. 总体架构

系统采用“前端 + C++ 业务后端 + ROS 适配层 + 本地存储”的四层结构。

### 3.1 前端层

前端为网页控制台，负责：

- 显示地图与机器人实时位置
- 显示点位、轨迹、当前任务状态
- 执行建图、点位管理、任务启动与暂停等操作
- 配置调度策略和系统参数
- 查看系统日志和告警

前端不直接拼装 ROS service 请求体，也不直接面向 topic/service 名称。

### 3.2 C++ 业务后端

后端是系统核心，负责：

- HTTP API 服务
- WebSocket 实时推送
- 任务调度与计划执行
- 自动回充状态机
- 参数配置管理
- 点位和地图业务管理
- 任务历史与日志存储
- 异常检测与恢复逻辑

### 3.3 ROS 适配层

ROS 适配层独立封装所有机器人通信能力，向业务后端暴露统一能力接口，不暴露底层协议细节。它需要同时支持两种运行方式：

- 上位机模式：通过 rosbridge/websocket 与机器人通信
- 本机模式：通过本地 ROS 客户端直接通信

两种方式实现统一抽象接口，供业务后端调用。

### 3.4 存储层

存储层采用 SQLite 保存业务配置和运行记录，包括：

- 系统参数
- 地图记录
- 充电点与投喂点
- 定时计划
- 任务执行记录
- 事件日志
- 恢复上下文

## 6. 部署设计

系统需要兼容两种部署模式。

### 6.1 上位机部署模式（优先）

- 前端与后端运行在上位机/工控机
- 后端通过 rosbridge 与机器人通信
- 优点是维护方便、升级灵活、适合现场调试

### 6.2 机器人本机部署模式

- 前端与后端运行在机器人服务器本机
- 后端可直接使用本地 ROS 客户端
- 优点是链路更短、外部依赖更少

这两种模式在业务逻辑上保持一致，仅在 ROS 适配层实现不同通信后端。

## 8. 任务状态机设计

### 8.1 核心状态

- `Idle`
- `Mapping`
- `Ready`
- `NavigatingToFeedPoint`
- `FeedingPause`
- `NavigatingToCharge`
- `Charging`
- `Paused`
- `Error`

### 8.2 核心事件

- 手动启动任务
- 定时任务触发
- 到达投喂点
- 停留结束
- 低电量触发
- 充电完成
- 定位异常
- ROS 连接断开
- 导航超时/失败
- 人工暂停
- 人工终止
- 立即回充

### 8.3 正常任务流程

1. 空闲等待手动或定时触发
2. 执行前检查
3. 前往第一个投喂点
4. 到点停留固定时长
5. 前往下一个投喂点
6. 所有点位完成后执行结束策略

### 8.4 回充中断流程

1. 任务进行中检测到电量低于阈值
2. 保存当前任务上下文
3. 切换到返航充电状态
4. 导航到充电点
5. 进入充电状态
6. 达到恢复条件后：
   - 若配置为继续任务，则恢复未完成任务
   - 若配置为等待人工，则进入待命状态

### 8.5 任务启动前检查

任务启动前必须验证：

- ROS 在线
- 地图已加载
- 充电点已配置
- 至少存在一个投喂点
- 当前电量高于启动阈值
- 定位状态正常

任一条件不满足，则不允许启动任务，并向前端返回明确错误原因。

## 9. 异常处理设计

### 9.1 ROS 连接断开

- 暂停任务推进
- 禁止下发新导航命令
- 自动重连
- 重连后重新评估当前状态，不盲目自动继续

### 9.2 定位丢失

- 立即暂停自动导航
- 标记为高优先级异常
- 等待定位恢复与人工确认

### 9.3 导航失败或长时间未到点

- 设置每点导航超时
- 允许有限重试
- 超过重试次数后进入异常态
- 记录失败点位和原因

### 9.4 充电失败

- 到达充电点后若未进入充电状态或电量未上升，进入异常态
- 记录告警并等待人工处理

### 9.5 人工干预

支持人工：

- 暂停任务
- 恢复任务
- 终止任务
- 立即回充
- 手动导航到指定点

人工干预必须被记录，并明确是否保留当前任务上下文。

## 10. 电量策略

建议采用两个阈值。

### 10.1 启动阈值

- 用于判断是否允许启动新任务
- 低于阈值时禁止启动任务

### 10.2 回充阈值

- 任务执行中低于阈值则中断任务并返航充电

这种双阈值设计可避免“任务刚开始又立刻因低电量回充”的抖动问题。

- `battery_start_threshold`
- `feed_pause_seconds`
- `resume_after_charge`

### 11.2 maps

保存地图记录：

- `id`

- `created_at`
- `is_active`

### 11.3 points

统一保存充电点与投喂点：

- `id`

- `type` (`charge` / `feed`)
- `x`
- `y`
- `theta`
- `sort_order`
- `enabled`
- `created_at`

### 11.4 task_schedules

保存定时计划：

- `id`
- `name`
- `enabled`
- `cron_expr` 或等价定时表达
- `point_order_json`
- `created_at`

### 11.5 task_runs

保存任务执行记录：

- `id`
- `trigger_type` (`manual` / `schedule`)
- `status`
- `started_at`
- `finished_at`
- `last_point_id`
- `resume_context_json`
- `error_message`

### 11.6 event_logs

保存事件与告警日志：

- `id`
- `level`
- `category`
- `message`
- `created_at`

- `robot_status`

## 14. ROS 适配层接口建议

适配层对业务后端暴露统一机器人能力方法，例如：

- `connect()`
- `disconnect()`
- `is_connected()`
- `start_mapping()`
- `stop_mapping()`
- `save_map(name)`
- `load_map(id)`
- `get_map_snapshot()`
- `get_robot_pose()`
- `get_battery()`
- `get_navigation_status()`
- `navigate_to_pose(pose)`
- `stop_navigation()`
- `set_initial_pose(pose)`
- `is_charging()`

业务后端只调用这些统一能力，不直接处理底层 ROS 数据格式细节。

- 异步与定时：asio

### 15.2 前端

- HTML/CSS/JavaScript

- 通过 REST + WebSocket 与后端通信

### 15.3 存储

- SQLite 保存业务数据
- 地图文件由 ROS 自身管理，业务层保存引用与元数据

## 16. 关键设计原则

- 前端不直接调用 ROS 原始接口
- 后端统一实现业务规则
- ROS 适配层只负责机器人能力封装
- 状态机统一管理任务过程
- 地图显示与机器人实时位置是主界面核心
- 低电量回充必须保存恢复上下文
- 异常状态必须明确可解释，不允许静默失败
